## Routing follow-ups

Generated by the route-planner work that landed on this branch. The planner is in `src/services/landRouting.js` (rasterization + A* + smoother) and the assistant entry points are in `src/services/assistant/tools/waterRouting.js`.

- **Time-aware CPA-based AIS avoidance** — the current `avoid_ais` flag projects each vessel forward along its course/speed and treats the swept corridor as a static obstacle. This is a first-order approximation: it bounds the area the vessel can be in, but doesn't model *when*. Proper CPA planning would compute the closest point of approach between the route (assuming a traveler speed) and each vessel's projected track, and replan when CPA falls below a threshold. Significantly bigger lift — temporal A* in space-time, or iterative replanning. Worth doing if the corridor approach turns out to be too conservative for fast traffic or too loose for slow.
- **Higher-detail coastline data for sub-km routing** — the planner uses Natural Earth 10m (~10 MB), which generalizes the coastline by 250-500 m. Fine for ocean-crossing and large-bay routes; visibly wrong at coastal/city scales. Three real upgrade paths, in increasing order of effort: (a) regional cut of OSM full-detail land polygons (works only in the cut region, ~30-50 MB), (b) GSHHG `h` shipped globally (~150-200 MB), (c) PMTiles-based on-demand loader. We tried OSM `simplified-land-polygons-complete` and it didn't actually help (still targeted at zoom 0-9) — that bullet is already done.
- **Route via with entry/exit hints** — `via_feature_ids` adds one waypoint per shape, at the shape's center. For prompts like *"route through Polygon X entering from the south and exiting at the north"* the agent currently falls back to `map_draw_route` with raw waypoints it computes itself. Could extend the via input to accept per-shape entry/exit bearings or sides, and have the tool pick boundary points instead of the centroid.
- **True polygon centroid for concave shapes** — `shapeCenter` in `tools/waterRouting.js` falls back to bbox midpoint for polygons. For L-shaped or donut polygons that's outside the polygon. Replace with a proper centroid (or "point-on-surface" technique that always lands inside).
- **CoT and manual-track avoidance in routes** — same projection logic as AIS, but currently only AIS is plumbed into `avoid_ais`. Manual tracks have course/speed too; CoT tracks do as well. Add `avoid_cot` / `avoid_manual_tracks` flags or unify into a single `avoid_moving_tracks: { ais, cot, manual }`. Reuses `vesselObstaclePolygon` once it's renamed.
- **Coastline accuracy review at common scales** — empirical sweep: try the planner at several typical zoom levels (ocean / regional / coastal city) with the current LAND_BUFFER_DEG and see where it visually clips. Tune the buffer / consider per-bbox scale dynamic buffer.

## Assistant coverage gaps

Map functionality the operator has in the UI but the assistant cannot drive. Each would be a small tool bundle modelled on the existing `perimeterTools` / `annotationTools` shape (`readonly` + confirm flow via `previewRender`). Grouped by likely impact.

- **Intercept tools** — listed separately below under "Intercept v1 follow-ups". Biggest gap: the assistant can set up a perimeter or a bloodhound but not an intercept. Model on `useMapIntercepts.interceptApi` (`addIntercept({ hostile, friendly, mode, offsetRange?, offsetBearing? })`, `removeIntercept(id)`, `setAimRingRadius(r)`, `clearAll()`). Tool surface: `intercept_list`, `intercept_add`, `intercept_remove`, `intercept_set_aim_radius`, `intercept_clear`.

- **Outbound CoT send** — no programmatic path at all today (the existing `scripts/cot-sender.mjs` is a dev script, not a service). Needs a service layer first — a Rust command + `useCotSender` composable — before there's anywhere for an agent tool to hook in. Listener CRUD has shipped; outbound send is the remaining gap.

- **Scenes: update / delete / apply** — `scenesTools` covers `scenes_list` + `scenes_create_scene`. The store exposes `updateScene` (label / description / icon), `deleteScene`, and `saveSceneCards`; none are surfaced. "Load/apply a scene" has no semantics today either — if scenes are meant to be recallable, that's a store design decision before it's a tool.

- **Measure (multi-segment chain)** — `map_measure_distance` handles a single pair of coordinates. The interactive measure tool in `useMapMeasure` accumulates a polyline chain of clicks; no tool surface for "total the route through these five points" beyond what `map_draw_route` would achieve. Low priority — probably subsumed by `map_draw_route` + `route_get` for most use cases.

- **Alerts — programmatic read** — `useMapAlerts` aggregates perimeter breaches (and will pick up more sources). No tool to list active alerts, get a specific one, or acknowledge. Natural follow-up once more sources are wired per the "More alert-chip sources" item below.

- **Manual-track editing coverage** — `map_create_track` creates them; `map_update_track` / `map_move_feature` / `map_rename_feature` / `map_update_feature_color` / `map_delete_feature` cover the edit surface. No obvious gap, but worth re-verifying once affiliation-filter or entity-type changes land.
